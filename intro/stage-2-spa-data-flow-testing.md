# 阶段 2 SPA 数据流改造测试说明

## 本阶段变更概览

阶段 2 解决的是 SPA 的数据流基础设施，不是业务等价重构。它把页面里“加载数据、提交操作、局部刷新、错误提示、URL 参数读取”收敛成统一模式，目标是让页面操作后只更新受影响的数据块，而不是整页刷新。

本阶段完成的关键变化：

- `useAsync` 升级为可重载、可忽略过期请求的资源钩子。
- 新增统一 mutation 处理方式，用于提交、确认、接单、已读、投票、保存等操作。
- 业务页面中的 `window.location.reload()` 已移除。
- 业务页面中的 `window.location.href` 硬跳转已改为 `Link` / `useNavigate`。
- URL query 读取统一收敛，不再在页面里散落 `new URLSearchParams(window.location.search)`。
- 消息、通知、订单、任务、纠纷、陪审、个人中心、管理员系统设置等页面已改成局部刷新。
- 阶段 2 验收脚本新增了组件测试和源代码约束检查。

## 自动化验证

在项目根目录执行：

```powershell
npm run typecheck
npm run build
npm run test:component
npm run test:stage02
npm run test:frontend-build
```

预期结果：

- 所有命令退出码为 0。
- `test:component` 应通过 SPA 数据流相关测试。
- `test:stage02` 应确认页面文件存在、组件测试通过、构建通过，并且业务页面没有整页刷新和硬跳转。
- `test:frontend-build` 应继续通过，并保持阶段 2 的源代码约束不回退。

## 手工验证环境

先构建前端产物：

```powershell
npm run build
```

启动本地开发环境：

```powershell
npm run dev
```

默认访问地址：

- 后端：`http://127.0.0.1:3001`

## 手工验收清单

### 1. 消息发送后局部刷新

进入消息中心，发送一条私信。

预期结果：

- 表单不整页刷新。
- 消息发送成功后，列表局部更新或重新拉取当前列表。
- 页面上不出现浏览器整页跳转。

### 2. 通知全部已读

进入通知中心，点击“全部已读”。

预期结果：

- 当前通知列表局部更新。
- 未读计数和摘要同步变化。
- 页面不刷新整页。

### 3. 订单确认完成

进入订单详情，点击“确认完成”。

预期结果：

- 仍停留在当前订单详情页。
- 当前订单状态局部更新。
- 不出现 `window.location.reload()` 造成的整页刷新感。

### 4. 任务接单与评论

进入任务详情，执行接单和发表评论。

预期结果：

- 接单成功后，详情状态更新。
- 评论提交后，评论区或详情区局部重拉。
- 页面不跳回整页。

### 5. 纠纷发起与陪审投票

进入纠纷发起页和陪审投票页，完成提交。

预期结果：

- 发起纠纷后返回订单列表或对应页的 SPA 导航。
- 陪审投票后仍留在当前路由，并刷新当前案件数据。

### 6. 个人中心与设置

在个人中心修改头像，进入设置页保存配置，切换钱包/订单/信用入口。

预期结果：

- 头像更新后只重拉当前用户资料。
- 设置保存后只重拉设置数据。
- 钱包和其他入口保持 SPA 内导航。

## 代码级回归检查

测试人员可以辅助确认以下约束仍成立：

- `frontend/src/spa/pages` 中不应再出现业务用途的 `window.location.reload()`。
- `frontend/src/spa/pages` 中不应再出现业务用途的 `window.location.href =`。
- 页面 query 读取应集中在统一 helper 或 React Router 参数读取里。

## 测试结论记录建议

建议在缺陷单里记录：

- 当前页面和操作步骤。
- 是否发生整页刷新。
- 当前数据是否只局部更新。
- 是否有错误提示被正确显示。
- 控制台和 Network 面板里是否出现异常的 document 请求。

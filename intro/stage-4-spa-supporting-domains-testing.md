# 阶段 4 SPA 辅助业务域测试说明

## 本阶段变更概览

阶段 4 的目标是把消息、通知、钱包、冻结明细、纠纷、陪审和信用详情从简化页面补齐为可验收的 React SPA 体验。前 3 个阶段已经覆盖运行时、数据流和用户核心交易闭环，本阶段继续围绕交易后的辅助业务能力做生产等价迁移。

本阶段完成的关键变化：

- 消息中心升级为左右结构，左侧展示会话列表，右侧展示当前会话、消息内容和发送区。
- 消息中心支持 `/messages?userId=...` 预填收件人，发送消息后只刷新当前消息资源，不整页刷新。
- 通知中心新增类型筛选、已读状态筛选、分页、单条标为已读和全部已读。
- 通知中的业务入口改为 React Router 站内跳转，避免硬跳转。
- 钱包页新增余额、冻结金额、可用余额、收入和支出摘要。
- 钱包交易列表支持 `all/income/expense/freeze/release/refund` 类型筛选、分页和关联订单/纠纷跳转。
- 冻结明细页支持 `all/active/dispute/released` 状态筛选，以及 `all/order/dispute` 原因筛选。
- 冻结记录新增释放条件、关联业务、创建/更新时间等时间线信息。
- 纠纷发起页支持从 `/disputes/new?order=...` 读取订单，也支持手填订单 ID。
- 纠纷证据提交统一为证据类型、说明和附件元数据，文件上传后会带入附件信息。
- 纠纷详情页补齐双方信息、订单/冻结信息、状态进度、证据列表、陪审统计和追加证据表单。
- 纠纷详情页提供“我的纠纷”入口区块，支持按状态、角色和分页查看。
- 陪审大厅展示可参与纠纷，并可进入对应投票页。
- 陪审投票页展示双方主张、证据、当前统计、历史投票和已投状态。
- 陪审投票值改为后端有效值：`publisher`、`provider`、`mediate`。
- 信用详情继续保留评分、分布、评价记录和规则说明，作为阶段 4 回归项。
- 阶段 4 页面继续遵守 SPA 数据流约束，不新增业务用途的整页刷新、硬跳转或散落 query 解析。
- `scripts/validate-stage-04.mjs` 已替换为阶段 4 专用验收脚本，覆盖静态约束、API 闭环、组件测试、类型检查和前端构建。

## 涉及页面与路由

测试人员重点关注以下路由：

- `/messages`
- `/messages?userId=1002`
- `/notifications`
- `/wallet`
- `/wallet/freeze`
- `/disputes/new`
- `/disputes/new?order=订单ID`
- `/disputes/:id`
- `/jury`
- `/jury/disputes/:id`
- `/users/:id/credit`

常用测试账号：

- 普通用户 A：`user_a` / `user123456`
- 普通用户 B：`user_b` / `user123456`

如果本地数据不是种子数据，纠纷、订单和冻结记录可能需要先通过发布需求、接单、确认或发起纠纷流程构造。

## 手工验证环境

先构建前端产物：

```powershell
npm run build
```

启动 SPA 模式本地服务：

```powershell
$env:FRONTEND_MODE="spa"
npm run dev
```

默认访问地址：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:3001`

## 手工验收清单

### 1. 消息中心

使用 `user_a` 登录后访问 `/messages?userId=1002`。

预期结果：

- 页面进入 React SPA 的消息中心。
- 左侧展示会话列表、未读数、最后消息预览和更新时间。
- 右侧收件人已预填为 `1002`。
- 填写消息内容并发送后，当前会话局部刷新。
- 页面不发生整页刷新，也不跳出当前 SPA。
- 搜索关键词后，URL query 和会话列表同步更新。

### 2. 通知中心

访问 `/notifications`，切换通知类型和已读状态筛选。

预期结果：

- 类型筛选包含全部、系统、订单、钱包、评价、纠纷、AI、社交。
- 已读筛选包含全部、已读、未读。
- 筛选和分页写入 URL，刷新页面后筛选状态仍保留。
- 点击单条“标为已读”后，只刷新通知列表和摘要。
- 点击“全部已读”后，未读统计更新。
- 通知业务入口只能跳到站内路径，不应出现外站跳转或浏览器硬跳转。

### 3. 钱包交易

访问 `/wallet`，切换交易类型筛选并翻页。

预期结果：

- 页面展示余额、冻结、可用余额、收入摘要和支出摘要。
- 交易类型筛选包含全部、收入、支出、冻结、释放、退款。
- 筛选和页码写入 URL，刷新后仍保持当前条件。
- 每条交易展示金额、类型、状态、原因、时间和关联业务。
- 点击“查看关联业务”能进入 `/orders/:id` 或 `/disputes/:id`。
- 页面切换筛选和翻页时不整页刷新。

### 4. 冻结明细

访问 `/wallet/freeze`，切换冻结状态和原因类型筛选。

预期结果：

- 状态筛选包含全部、生效中、纠纷中、已释放。
- 原因筛选包含全部、订单、纠纷。
- 每条冻结记录展示金额、状态、原因、释放条件和关联标题。
- 时间线展示创建、更新和释放相关节点。
- 点击“查看关联业务”能进入对应订单或纠纷详情。
- 筛选和分页刷新后仍能恢复。

### 5. 发起纠纷

从订单详情的纠纷入口进入，或直接访问 `/disputes/new?order=订单ID`。

预期结果：

- 页面能读取 URL 中的订单 ID。
- 也可以手动填写订单 ID。
- 提交 payload 应包含类型、原因、描述和初始证据。
- 上传附件后，证据列表中应包含文件名、类型、大小等附件元数据。
- 提交成功后进入纠纷详情或显示明确成功状态。

### 6. 纠纷详情与补充证据

访问 `/disputes/:id`。

预期结果：

- 页面展示纠纷状态、双方用户、订单信息、冻结信息和进度时间线。
- 页面展示已有证据列表。
- 填写证据说明并提交后，证据列表局部刷新，新增证据可见。
- 我的纠纷入口区块可按状态、角色和分页查看。
- 操作过程中不发生整页刷新。

### 7. 陪审大厅与投票

使用 `user_b` 登录后访问 `/jury`，进入某个 `/jury/disputes/:id`。

预期结果：

- 陪审大厅展示可参与纠纷、状态、原因和创建时间。
- 投票页展示双方主张、双方证据、当前票数统计和历史投票。
- 投票表单只有三个有效方向：需求方胜诉、服务方胜诉、建议调解。
- 实际提交值应对应 `publisher`、`provider`、`mediate`。
- 必须填写投票理由才能提交。
- 投票成功后页面展示“已投票”，并禁用重复投票。
- 如果账号已经投过票，页面直接展示已投方向和理由。

### 8. 信用详情

访问 `/users/:id/credit`。

预期结果：

- 页面展示信用评分、等级或分布信息。
- 页面展示评价记录。
- 刷新当前地址后仍能恢复信用详情页面。
- 从评价、用户信息或其他入口进入时仍保持 SPA 内导航。

## 代码级回归检查

测试人员可以辅助确认以下约束仍成立：

- 阶段 4 页面不应出现业务用途的 `window.location.reload()`。
- 阶段 4 页面不应出现业务用途的 `window.location.href =`。
- 阶段 4 页面不应散落使用 `new URLSearchParams(window.location.search)`。
- 筛选、分页和详情跳转应通过 React Router、共享 helper 或统一 query 工具完成。
- 提交消息、通知已读、证据提交和陪审投票后，应局部刷新当前资源。

## 缺陷记录建议

提交缺陷时建议记录：

- 使用账号和当前路由。
- 操作前的 URL query。
- 操作步骤和预期结果。
- 是否发生整页刷新或新的 `document` 请求。
- Network 面板中对应 API 的请求和响应。
- 消息、通知、钱包、冻结、纠纷或陪审数据是否局部更新。
- 如果是陪审问题，记录投票方向、理由和是否已投过票。
- 如果是证据问题，记录上传文件名、大小、类型和证据说明。

# 阶段 5 SPA AI 与治理后台测试说明

## 本阶段变更概览

阶段 5 的目标是把 AI 用户侧能力和 AI 管理治理能力从简化页面升级为生产等价的 React SPA 体验。阶段 4 已经完成消息、通知、钱包、纠纷、陪审等辅助业务域，本阶段继续迁移 AI 助手、AI 筛选结果，以及管理员侧 AI 日志、会话、反馈、异常和配置页面。

本阶段完成的关键变化：

- AI 助手页升级为完整会话体验，支持历史会话列表、场景切换、新对话、流式回答和局部刷新。
- AI 助手支持从 URL query 读取 `scene`、`prompt`、`conversationId`，刷新页面后能恢复当前上下文。
- AI 回答支持富文本渲染、复制、用户反馈，以及跳转到 AI 筛选结果或发布草稿入口。
- 用户壳层新增固定 AI 入口，可从登录后的用户页面快速进入 `/ai/assistant`。
- AI 筛选结果页使用 `/ai/results?prompt=...` 驱动筛选条件，展示解析条件、推荐需求、匹配原因、评分和站内详情链接。
- 管理员 AI 路由从通用占位页切换为明确 React 页面。
- AI 日志页支持关键字、用户、会话、场景、状态筛选，展示调用耗时、token、风险级别和时间。
- AI 会话管理页支持筛选、分页、会话列表和右侧会话详情，详情内容保留脱敏说明。
- AI 用户反馈页支持筛选、单条处理、批量处理和反馈报告生成。
- AI 异常调用页支持异常筛选、选择异常记录、重试低风险失败和创建内部事件单。
- AI 配置页支持启用状态、模型、频控、上下文、安全阈值、场景开关、敏感词检测和安全边界展示。
- 阶段 5 页面继续遵守 SPA 数据流约束，不新增业务用途的整页刷新、硬跳转或旧原型 AI 弹窗依赖。
- `scripts/validate-stage-05.mjs` 已替换为阶段 5 专用验收脚本，覆盖静态约束、AI API 闭环、组件测试、类型检查和前端构建。

## 涉及页面与路由

测试人员重点关注以下用户侧路由：

- `/ai/assistant`
- `/ai/assistant?scene=request_filter&prompt=找一个信用高的英语辅导需求`
- `/ai/assistant?conversationId=会话ID`
- `/ai/results`
- `/ai/results?prompt=找一个信用高的英语辅导需求`

测试人员重点关注以下管理侧路由：

- `/admin/ai/logs`
- `/admin/ai/conversations`

- `/admin/ai/errors`
- `/admin/ai/config`

## 自动化验证

在项目根目录执行：

```powershell
npm run typecheck
npm run test:component
npm run test:stage05
npx playwright test tests/e2e/spa-stage-05.spec.ts
npm run test:e2e
npm run test:frontend-build
```

预期结果：

- 所有命令退出码为 0。
- `test:stage05` 应通过阶段 5 静态检查、AI 用户 API 闭环、AI 管理 API 闭环、组件测试、类型检查和前端构建。
- `test:component` 应包含阶段 5 组件测试，覆盖 AI 助手、AI 筛选结果、AI 日志、AI 反馈、AI 异常和 AI 配置。
- `tests/e2e/spa-stage-05.spec.ts` 应覆盖真实 SPA 登录后的 AI 问答、反馈、AI 筛选结果，以及管理员 AI 治理页 hydrate 和核心操作。
- `test:e2e` 应包含阶段 5 e2e，并继续通过阶段 3、阶段 4 和生产运行时回归。
- `test:frontend-build` 应确认 SPA 构建产物、路由 fallback 和前端约束没有回退。

说明：构建过程中如果只出现 Vite chunk 体积警告，但命令退出码为 0，则不视为阶段 5 验收失败。

## 手工验证环境

先构建前端产物：

```powershell
npm run build
```

启动 SPA 模式本地服务：

```powershell
$env:FRONTEND_MODE="spa"
npm run dev
```

默认访问地址：

- 前端：`http://127.0.0.1:5173`
- 后端：`http://127.0.0.1:3001`

## 手工验收清单

### 1. 用户侧 AI 入口

使用 `user_a` 登录后进入任意用户壳层页面，例如 `/feed` 或 `/messages`。

预期结果：

- 顶部用户导航区域能看到 AI 入口。
- 点击 AI 入口后进入 `/ai/assistant`。
- 页面保持 SPA 导航，不出现整页硬跳转。
- HTML 路由标识应为 `data-route-id="ai-assistant"`。

### 2. AI 助手基础问答

访问 `/ai/assistant`。

预期结果：

- 页面展示 AI 助手标题、场景切换、历史会话、对话窗口和输入区。
- 场景包含全部、筛选、发布、规则、摘要。
- 输入问题并点击“发送”后，页面出现用户消息和 AI 回复。
- AI 回复应在当前对话中局部更新，不应整页刷新。
- 发送后历史会话列表或最近会话区域会刷新。
- 请求失败时应在当前回答位置展示错误提示，而不是空白页。

### 3. AI 助手 URL 恢复

访问：

```text
/ai/assistant?scene=request_filter&prompt=帮我找一个信用高的英语辅导需求
```

预期结果：

- 当前场景切到筛选相关状态。
- 输入框中预填 URL 中的 prompt。
- 刷新页面后场景和 prompt 仍能恢复。
- 如果使用 `/ai/assistant?conversationId=会话ID`，应能定位到对应历史会话并加载消息。

### 4. AI 回答操作

在 AI 助手页发送问题并等待 AI 回复。

预期结果：

- 点击“复制”可复制 AI 回复文本。
- 点击“有用”或“无用”后，按钮状态变为已反馈状态。
- 反馈提交只调用当前消息反馈接口，不刷新整页。
- 点击“查看匹配结果”应进入 `/ai/results?prompt=...`。
- 点击“生成草稿”应进入发布页，并通过 query 携带草稿内容。

### 5. AI 筛选结果

访问：

```text
/ai/results?prompt=找一个信用高的英语辅导需求
```

预期结果：

- 页面展示筛选描述输入区。
- 页面展示 AI 解析出的条件。
- 页面展示推荐需求列表。
- 每条推荐展示标题、分类、发布者、信誉、时间币、匹配分和匹配原因。
- “查看详情”只跳转到站内路径，例如 `/posts/:id`。
- 修改筛选描述并点击更新或筛选后，URL query 和结果同步更新。
- 刷新当前地址后仍能根据 query 恢复结果。

### 6. AI 日志管理

使用 `admin_main` 登录后访问 `/admin/ai/logs`。

预期结果：

- 页面进入管理员壳层，路由标识为 `admin-ai-logs`。
- 页面展示 AI 日志管理标题、AI 管理子导航、摘要卡片和日志表格。
- 可按关键字、用户 ID、会话 ID、场景和状态筛选。
- 筛选条件应写入 URL query。
- 表格展示会话、用户、场景、状态、耗时、token、风险和时间。
- 翻页时只刷新日志数据，不整页刷新。

### 7. AI 会话管理

访问 `/admin/ai/conversations`。

预期结果：

- 页面展示会话列表和会话详情区域。
- 可按关键字、用户、会话、场景和状态筛选。
- 点击会话行后，URL 中带上 `conversationId`，右侧详情加载对应会话。
- 详情中展示会话状态、消息数、脱敏说明和消息内容。
- 页面不应显示未脱敏的敏感字段，例如 password、token、secret、api_key、phone。

### 8. AI 用户反馈

访问 `/admin/ai/feedback`。

预期结果：

- 页面展示反馈列表、筛选区、批量处理按钮和生成周报按钮。
- 可按关键字、用户、会话、场景、状态和反馈类型筛选。
- 点击单条“标记已处理”后，该反馈进入已处理状态或列表刷新。
- 勾选反馈后点击“批量处理”，应调用批量处理能力并刷新列表。
- 点击“生成周报”后不离开当前页面，成功后列表或状态保持可用。
- 点击“查看会话”应带着对应 `conversationId` 进入会话管理上下文。

### 9. AI 异常调用

访问 `/admin/ai/errors`。

预期结果：

- 页面展示异常调用列表、筛选区、重试低风险失败和创建事件单按钮。
- 可按关键字、用户、会话、场景、状态和异常类型筛选。
- 勾选异常记录后，“重试低风险失败”可用，并将符合条件的记录加入内部重试队列。
- 勾选异常记录后，“创建事件单”可用，并成功创建内部事件单。
- 操作完成后只刷新异常列表，不整页刷新。
- 高风险或不可重试异常不应被错误地当作可自动重试成功处理。

### 10. AI 配置管理

访问 `/admin/ai/config`。

预期结果：

- 页面展示启用 AI、模型、频控、并发、上下文、安全阈值、超时、最大 token、温度、检测模式、告警阈值和会话保留天数。
- 页面展示场景开关，例如 chat、request_filter 等。
- 页面展示安全边界，包括 AI 可做、AI 不可做和需审计事项。
- 修改配置并点击“保存配置”后，页面局部保存并刷新配置。
- 初始场景开关保存时不应丢失已有场景配置。
- 刷新页面后，最新配置仍能从后端读取。

## 代码级回归检查

测试人员可以辅助确认以下约束仍成立：

- 阶段 5 页面不应出现业务用途的 `window.location.reload()`。
- 阶段 5 页面不应出现业务用途的 `window.location.href =`。
- 阶段 5 页面不应依赖旧原型 AI 弹窗脚本。
- 用户侧 AI 会话、反馈、筛选和管理侧操作都应使用当前 SPA API client。
- 管理侧 `/admin/ai/*` 路由不应再落回 `AdminGenericPage` 通用占位页。
- AI 筛选结果和管理侧筛选分页应通过 URL query 保持可恢复。
- 站内业务跳转应使用 React Router `Link` 或安全站内 href helper。

## 缺陷记录建议

提交缺陷时建议记录：

- 使用账号和当前路由。
- 操作前后的 URL query。
- 选择的 AI 场景、输入 prompt 和 conversationId。
- Network 面板中对应 API 的请求与响应。
- AI 回复是否流式出现，是否最终落入历史会话。
- 反馈按钮点击后对应 messageId 和反馈值。
- 管理侧筛选条件、页码、选中记录 ID。
- 配置保存前后的字段值，特别是场景开关。
- 是否发生整页刷新、硬跳转或旧原型弹窗。
- 如果是异常治理问题，记录 callId、异常类型、风险级别和操作按钮。

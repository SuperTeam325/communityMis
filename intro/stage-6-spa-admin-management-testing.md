# 阶段 6 SPA 管理后台测试说明

## 本阶段变更概览

阶段 6 的目标是把管理后台从通用占位页升级为生产等价的 React SPA 管理台。前 5 个阶段已经覆盖运行时、数据流、用户核心闭环、辅助业务域和 AI 治理，本阶段继续把后台治理能力补齐到可验收状态。

本阶段完成的关键变化：

- 管理仪表盘、用户管理、交易流水、争议处理、纠纷终审、平台统计、标签/类别管理、敏感词管理、内容风险审核、审计日志、系统设置和备份都切成了明确的 React 页面。
- `AdminGenericPage` 不再作为后台生产路由的默认落点，后台路由现在按页面组件明确分发。
- 管理端侧边栏高亮、页面标题和基础导航继续由路由元信息驱动。
- 高风险操作统一使用 React 确认弹窗，不再依赖原生 `window.prompt`。
- 系统页支持系统配置、备份创建、备份恢复、备份删除和消息清理预览/执行。
- 纠纷终审页支持从 `/admin/disputes/final?disputeId=...` 直达，并按后端有效值提交 `publisher_win`、`provider_win`、`mediate`。
- 内容风险页支持单条处置和批量进入人工复核，批量动作的语义是“进入复核”，不是批量通过/拒绝。
- 标签/类别、敏感词页面当前以筛选、列表、状态切换和导入为主；后端未开放删除路由，因此验证以停用/禁用为准。
- `scripts/validate-stage-06.mjs` 已替换为阶段 6 专用验收脚本，覆盖静态约束、后台 API 闭环、组件测试、类型检查和前端构建。

## 自动化验证

在项目根目录执行：

```powershell
npm run typecheck
npm run test:component
npm run test:stage06
npx playwright test tests/e2e/spa-stage-06.spec.ts
npm run test:e2e
npm run test:frontend-build
```

预期结果：

- 所有命令退出码为 0。
- `test:stage06` 应通过阶段 6 静态检查、后台 API 闭环、类型检查、组件测试和前端构建。
- `test:component` 应包含阶段 6 组件测试，覆盖用户、交易、争议、敏感词、风险内容、审计和系统页。
- `tests/e2e/spa-stage-06.spec.ts` 应覆盖真实 SPA 登录后的管理后台 hydrate 和核心页面。
- `test:e2e` 应继续通过阶段 3、阶段 4、阶段 5、阶段 6 和生产运行时回归。
- `test:frontend-build` 应确认 SPA 构建产物、路由 fallback 和前端约束没有回退。

说明：构建过程中如果只出现 Vite chunk 体积警告，但命令退出码为 0，则不视为阶段 6 验收失败。

## 手工验收清单

### 1. 管理登录与壳层

使用 `admin_main` 登录后访问 `/admin/dashboard`。

预期结果：

- 页面进入管理后台壳层。
- 左侧导航高亮当前模块。
- 顶部标题与当前页面标题一致。
- HTML 路由标识应为 `data-route-id="admin-dashboard"`。
- 不应再落入通用 `AdminGenericPage` 占位页。

### 2. 仪表盘

访问 `/admin/dashboard`。

预期结果：

- 页面展示管理仪表盘标题和关键指标。
- 能进入用户、交易、争议、内容风险和系统设置等快捷入口。
- 页面数据应通过后端管理 API 加载，不是静态占位内容。

### 3. 用户管理

访问 `/admin/users`。

预期结果：

- 页面展示用户列表、状态、信用和创建时间。
- 可按状态、最低信用、最高信用和关键词筛选。
- 筛选条件应写入 URL query，刷新后仍能恢复。
- 点击启用/禁用后应调用状态更新接口并局部刷新列表。
- 不应出现整页刷新。

### 4. 交易流水

访问 `/admin/transactions`。

预期结果：

- 页面展示交易列表、金额、余额和关联业务。
- 可按类型、用户 ID、订单 ID 和关键词筛选。
- 交易类型应包含收入、支出、冻结、解冻、退款和平台费用。
- 关联订单或纠纷链接应跳转到站内路径。
- 翻页时只刷新当前列表。

### 5. 争议处理与终审

访问 `/admin/disputes`，再进入 `/admin/disputes/final`。

预期结果：

- 争议列表支持状态和关键词筛选。
- 点击列表项后，右侧详情面板显示争议、订单、描述和金额信息。
- 点击“终审”可进入终审页，并能从 `?disputeId=...` 直达。
- 终审结果只接受 `publisher_win`、`provider_win`、`mediate`。
- 终审提交需要理由，退款金额可填写或留空。
- 提交后只刷新当前纠纷数据，不整页刷新。

### 6. 标签/类别管理

访问 `/admin/categories`。

预期结果：

- 页面展示类别和标签列表。
- 可查看名称、编码、状态和排序信息。
- 当前实现以状态切换为主，验证时重点确认能切换为停用/禁用状态。
- 这类资源当前不使用删除按钮，后端删除路由未开放。

### 7. 敏感词管理

访问 `/admin/sensitive-words`。

预期结果：

- 页面展示敏感词列表、等级、类别、状态和替换词。
- 可按等级、状态和关键词筛选。
- 可执行批量导入。
- 每条敏感词可切换为停用/禁用状态。
- 确认弹窗应要求输入确认短语，不应使用原生 `prompt`。

### 8. 内容风险审核

访问 `/admin/risk-content`。

预期结果：

- 页面展示风险内容列表、状态、风险等级、来源和时间。
- 可按状态、风险等级、来源和关键词筛选。
- 选中多条后可执行“批量复核”。
- 单条操作支持复核、通过和移除。
- 批量动作的结果应是进入人工复核，不应被误认为最终通过或拒绝。

### 9. 审计日志

访问 `/admin/audit-log`。

预期结果：

- 页面可按操作者、动作、对象类型、对象 ID 和关键词筛选。
- 高风险动作应有明显标识，例如用户状态变更、纠纷终审、风险内容处置、系统设置和备份操作。
- 日志内容为只读，不应出现编辑入口。

### 10. 系统设置、备份与消息清理

访问 `/admin/system`。

预期结果：

- 可编辑冻结天数、自动归档天数、新用户赠币、维护模式、自动备份、AI 高风险阻断和安全公告。
- 创建备份需要输入确认短语“立即备份”和理由。
- 恢复备份需要输入确认短语“恢复备份”和理由。
- 删除备份需要输入确认短语“删除备份”和理由。
- 消息清理支持预览和执行两种模式。
- 执行清理时需要输入确认短语“清理归档消息”。
- 以上动作都应在当前页局部刷新，不整页刷新。

## 代码级回归检查

测试人员可以辅助确认以下约束仍成立：

- 阶段 6 页面不应出现业务用途的 `window.location.reload()`。
- 阶段 6 页面不应出现业务用途的 `window.location.href =`。
- 阶段 6 页面不应出现业务用途的 `window.prompt`。
- 阶段 6 页面不应散落使用 `new URLSearchParams(window.location.search)`。
- 管理后台路由不应再落到通用 `AdminGenericPage`。
- 筛选、分页、详情和确认弹窗都应使用当前 SPA 组件和路由查询参数完成。
- 业务链接应只跳转站内路径，不应出现外站硬跳转。

## 缺陷记录建议

提交缺陷时建议记录：

- 使用账号和当前路由。
- 操作前后的 URL query。
- 所选筛选条件、页码和操作对象 ID。
- Network 面板中对应 API 的请求与响应。
- 是否发生整页刷新、硬跳转或原生弹窗。
- 备份/终审/风险处置/系统设置时填写的确认短语和理由。
- 如果是内容风险问题，记录选中的风险 ID、处置类型和是否批量复核。
- 如果是纠纷终审问题，记录最终结果、退款金额和理由。
